如果你已经确定当前就是在推进 `platform-api`，优先从这里进入，而不是直接在 `docs/` 根层里找散落文档。

## 后续 step 文档默认结构

从当前开始，新的 step 文档统一按下面这套模板写。

要点先说清：

- 开发侧验收也放在服务器侧执行
- 测试侧验收也放在服务器侧执行
- 不能只写“验收结果”，还要写“怎么验”

### 标准模板

1. 这一步的目标
2. 预期结果
3. 这一步的代码设计
4. 函数调用流程图（包含各函数名）
5. 开发侧验收步骤（服务器侧执行）
6. 开发侧验收结果
7. 测试侧验收步骤（服务器侧执行）
8. 测试侧验收结果
9. 相关专题与测试文档

### 每一段应该写什么

#### 1. 这一步的目标

写清：

- 这一轮到底要补什么能力
- 不补会卡住后面什么
- 这一步的边界到哪里为止

#### 2. 预期结果

写清：

- 做完后系统应该具备哪些可观察结果
- 哪些接口 / 字段 / 行为应该出现
- 哪些结果这一步先不做

#### 3. 这一步的代码设计

写清：

- 这一轮涉及哪些层
- 每层负责什么
- 关键文件和关键函数分别承担什么职责

#### 4. 函数调用流程图（包含各函数名）

必须体现：

- 接口入口
- router 函数名
- service 函数名
- repository 函数名
- schema / response 的落点

优先用 `mermaid`，并尽量让人一眼看出调用顺序。

#### 5. 开发侧验收步骤（服务器侧执行）

这里不是写 pytest，而是写“这轮代码功能怎么在服务器上手动验”。

例如可以包括：

- 启动服务命令
- `curl` / API 调用命令
- 数据库文件 / 返回 JSON / 状态码检查方法
- 需要人工确认的功能点

#### 6. 开发侧验收结果

这里写“按上面的开发侧验收步骤执行后，应该确认哪些结果成立”。

推荐写成 checklist，例如：

- [x] 路由已可访问
- [x] 返回字段符合预期
- [x] 数据已写入 / 已查出
- [x] 功能行为与当前代码设计一致

#### 7. 测试侧验收步骤（服务器侧执行）

这里写自动化测试怎么验。

例如可以包括：

- `pytest` 命令
- `allure` 结果生成命令
- 需要跑哪些测试文件 / 测试集

#### 8. 测试侧验收结果

这里写测试跑完后要确认哪些结果。

例如：

- [x] 命中路径已覆盖
- [x] 未命中路径已覆盖
- [x] 一致性场景已覆盖
- [x] Allure 结果文件已产出

#### 9. 相关专题与测试文档

这里统一链接到：

- `guides/`
- 对应 step 的 `testing-training/`
- 对应 step 的 `testing-automation/`

### 推荐骨架

```markdown
## 这一步的目标

## 预期结果

## 这一步的代码设计

## 函数调用流程图

## 开发侧验收步骤（服务器侧执行）

## 开发侧验收结果

## 测试侧验收步骤（服务器侧执行）

## 测试侧验收结果

## 相关专题与测试文档
```

