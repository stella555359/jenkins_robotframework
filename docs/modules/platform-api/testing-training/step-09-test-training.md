# Step 9 Test Training

## 这份文档的目标

这不是业务实现文档，而是一轮专门围绕 `GET /api/runs/{run_id}` 做的测试设计训练记录。

它解决的是下面这些问题：

- 当前 `platform-api` 这几个 API 应该怎么系统地看测试覆盖
- Step 9 这一步最小但够用的测试集合到底是什么
- 哪些风险现在最值得优先盯
- 测试工程师和 AI 在这一步怎么配合

## 当前 API 测试矩阵

### `GET /api/health`

- 主要目的：确认健康检查契约
- 当前 happy path：`200 + 预期字段`
- 当前 negative path：暂时优先级低
- 是否需要持久化验证：不需要
- 当前状态：已覆盖
- 建议下一步：继续保持为轻量契约测试

### `POST /api/runs`

- 主要目的：创建 run
- 当前 happy path：创建成功并返回 `run_id / status / message`
- 当前 negative path：`run_id` 冲突重试
- 是否需要持久化验证：需要
- 当前状态：已覆盖
- 建议下一步：补请求校验边界场景

### `GET /api/runs`

- 主要目的：返回 run 列表
- 当前 happy path：最新创建的记录排在前面
- 当前 negative path：后续可补空列表场景
- 是否需要持久化验证：间接需要
- 当前状态：已覆盖
- 建议下一步：补零数据场景和后续分页结构预留

### `GET /api/runs/{run_id}`

- 主要目的：按 `run_id` 精确查详情
- 当前 happy path：返回正确记录
- 当前 negative path：查不到时返回 `404`
- 是否需要持久化验证：需要
- 当前状态：已覆盖
- 建议下一步：补列表和详情的一致性测试

## Step 9 最小测试集合

### 第 1 层：API 契约层

这一层至少要保证：

1. 已存在的 `run_id` 时，接口返回 `200`
2. 返回字段和 JSON 结构完整
3. 不存在的 `run_id` 时，明确返回 `404`

最短记忆版：

```text
先确认接口“说得对不对”。
```

### 第 2 层：持久化验证层

这一层至少要保证：

1. 先创建 run，再查详情
2. 详情返回内容和数据库真实记录一致
3. 后续逐步补列表与详情的一致性校验

最短记忆版：

```text
再确认接口不是“假查到”。
```

## 当前高优先级风险清单

### 风险 1：`404` 语义漂移

风险描述：

- 查不到时如果被改成 `200 + {}`
- 调用方会误判为“查到了，只是字段为空”

建议测试：

- 明确断言缺失 run 返回 `404`
- 明确断言错误消息稳定为 `{"detail": "Run not found."}`

优先级：高

### 风险 2：router 层返回假数据

风险描述：

- 如果 router 直接手写假的详情数据
- 前端看到的结果就不再代表数据库真实状态

建议测试：

- 继续坚持 DB-backed API 测试
- 不要让详情接口脱离真实 `runs` 表

优先级：高

### 风险 3：列表与详情不一致

风险描述：

- 列表里能看到某条 run
- 但详情页按同一 `run_id` 打不开

建议测试：

- 创建 run
- 先查列表
- 再查详情
- 确认同一 `run_id` 两边语义一致

优先级：中

### 风险 4：响应 schema 漂移

风险描述：

- 某次改动后字段静默变化
- 但接口行为表面看仍然“能用”

建议测试：

- 对当前关键字段做契约断言
- 不只断言状态码，也断言响应结构

优先级：中

## 推荐测试 workflow

以后碰到类似 Step 9 的接口，你可以优先按这个顺序做测试设计：

### Step A：先确认这一步的主语义

对 Step 9 来说，重点不是“把列表再查一遍”，而是：

```text
按 run_id 精确查单条记录。
```

### Step B：先列 happy path 和 failure path

不要一上来就写代码，先写清：

- 查到时应该怎样
- 查不到时应该怎样

### Step C：判断需不需要持久化验证

如果这一步只是纯计算，可能不需要查 DB。

但 Step 9 这一步很明显需要，因为它的真实性完全依赖数据库记录。

### Step D：再决定 fixture 要不要补

当前这一步最值得复用的 fixture 已经有：

- `client`
- `db_connection`
- `create_run_via_api`

### Step E：最后才写 pytest 和 Allure 标注

也就是说，测试代码应该是最后一步，不是第一步。

最短记忆版：

```text
先想语义，再拆路径，再看要不要查库，最后才写测试代码。
```

## AI 和测试工程师怎么配合

### AI 最适合先做的事情

1. 帮你列测试矩阵
2. 主动指出漏掉的失败场景
3. 帮你区分 API 契约层和持久化验证层
4. 先出题逼你复述调用链和测试设计

### 你自己最值得做的事情

1. 自己判断这一步的主语义
2. 自己判断哪些路径是高风险路径
3. 自己在服务器上执行 `pytest` / `Allure`
4. 自己在 review 时判断测试是不是“测得太假”

最短记忆版：

```text
AI 负责帮你看全和补漏，你负责做最后的风险判断和真实环境执行。
```

## 这一轮最值得继续练的下一题

下一轮最推荐继续补的训练点是：

```text
列表接口与详情接口的一致性测试设计
```

也就是：

1. 创建一条 run
2. 先在 `GET /api/runs` 里看到它
3. 再用同一个 `run_id` 调 `GET /api/runs/{run_id}`
4. 确认两边字段语义一致
