## 问题

用户想理解：不管是原始自动化平台还是当前轻量化方案，为什么都值得额外增加一层 React 门户层；以及这种做法和直接进入 Jenkins 手工触发 Job 相比，具体收益在哪里。

## 回答

门户层的核心价值不是“多一层更复杂”，而是把“给人使用的业务入口”和“给 Jenkins 执行的调度入口”分开。

相对于直接使用 Jenkins，门户层的主要好处包括：

- 更贴近业务语义：用户面对的是测试线、版本、用例、环境，而不是 Job、参数、Agent、构建号
- 更少误操作：可以通过下拉框、必填校验、默认值和参数组合限制减少填错
- 更清晰的安全边界：浏览器不直接接触 Jenkins 凭据、Token 和内部接口
- 更好的使用体验：更容易做任务触发页、构建历史页、结果详情页、趋势页
- 更容易平台化：可以统一接 Jenkins、结果库、报告、权限、通知等多个后端能力
- 更合理的分工：React 负责门户，FastAPI 负责 API 和聚合，Jenkins 专注调度和执行

如果目标只是“先跑起来”，直接 Jenkins 更快更简单；如果目标是“做成平台，并练 React + FastAPI”，门户层很有价值。
