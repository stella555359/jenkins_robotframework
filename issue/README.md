# issue

这个目录用于记录本项目实施过程中遇到的问题，以及对应的原因分析和解决方法。

## 命名建议

- 优先按问题类别归并记录，而不是每个小问题单独一个文件
- 同一主题的问题持续追加到同一个文件中
- 文件名建议体现“类别”而不是只体现某一次报错

示例：

- `git-issues.md`
- `jenkins-deploy-issues.md`
- `portal-runtime-issues.md`
- `workspace-collaboration-issues.md`

如果某类问题目前还不多，也可以先临时使用按日期命名，但后续同类问题增多时，建议整理合并到类别文件中。

## 记录结构建议

```md
# 标题

## 日期
2026-04-16

## 背景
描述当时正在做什么。

## 问题
描述具体问题，尽量附上原始报错。

## 原因
说明根因判断。

## 解决方法
记录最终修复步骤。

## 后续注意
记录后续检查项、注意事项或预防措施。
```

## 归档建议

- Git 相关问题优先统一记录到 `git-issues.md`
- Jenkins 部署和配置类问题优先统一记录到 `jenkins-deploy-issues.md`
- Portal 服务运行类问题优先统一记录到 `portal-runtime-issues.md`
- 协作约定、记录规范、Copilot 工作方式说明类问题优先统一记录到 `workspace-collaboration-issues.md`
- 如果已有同类文件，默认更新原文件，不再新建分散的 issue 文件

## 说明

- `issue/` 目录中的文件属于仓库内可见文档，会跟随 Git 提交和版本管理。
- 如果提到 Copilot 的“仓库记忆”，那是工作区级辅助记忆，不等同于当前仓库里的普通文件。
- 对用户可见、需要长期留档的规则和问题结论，仍应补充记录到 `issue/` 目录中。

后续这个仓库里遇到的问题和解决方法，统一记录在这里，并使用中文书写。
