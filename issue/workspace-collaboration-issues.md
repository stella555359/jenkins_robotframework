# 工作区协作与记录规范问题汇总

## 问题 1：Copilot 提到的“仓库记忆”存放在哪里

### 日期
2026-04-16

### 背景
在整理本仓库的 issue 记录规范时，提到了“我也把这条规则写进了仓库记忆”。随后需要进一步说明这个“仓库记忆”实际存放位置，以及它和仓库内普通 Markdown 文件的区别。

### 问题
“仓库记忆”是否就在当前仓库目录中，是否会跟随 Git 一起提交和同步？

### 原因
“仓库记忆”这个说法容易和仓库里的普通说明文档混淆。对于使用者来说，如果不区分这两类信息，就很容易误以为它和 `issue/` 下的 Markdown 文件一样，属于仓库本身的一部分。

### 解决方法
这里需要明确区分两套记录位置：

1. `issue/` 目录中的文件：属于仓库内可见文件，会跟随 Git 版本管理。
2. Copilot 的“仓库记忆”：属于工作区级辅助记忆，用于帮助后续继续遵守当前仓库约定，不是当前项目目录树中的普通文件。

因此：

- 需要给用户长期留档、需要团队可见、需要进入 Git 历史的内容，应写入 `issue/` 目录。
- 仅用于辅助 Copilot 在当前工作区持续遵守约定的信息，可以写入“仓库记忆”。
- 如果某条规则既需要我记住，也需要仓库内可见，则两边都应保留一份。

### 后续注意

1. 后续凡是涉及协作规则、记录规范、工作方式说明的问题，优先追加到本文件。
2. 对外可见的规则，不能只放在“仓库记忆”里，还要同步写入 `issue/README.md` 或相应 issue 汇总文件。

## 问题 2：实际做项目时如何分工使用 main、dev、feature、fix 分支，避免混乱

### 日期
2026-04-16

### 背景
文档中已经给出最小分支策略：`main`、`dev`、`feature/*`、`fix/*`。但在实际项目推进时，如果只有分支名称而没有具体分工规则，很容易出现分支长期不清理、多人共用同一分支、功能和修复混在一起的问题。

### 问题
这些分支在日常开发中应该怎么分工，才能既够用又不凌乱？

### 原因
分支混乱通常不是因为分支类型太多，而是因为缺少几个最基本的协作规则：

1. `main` 被直接拿来开发
2. 一个 feature 分支里同时塞多个无关需求
3. 功能分支长期不合并，越拖越难收口
4. 缺陷修复和新功能混在同一条分支里

### 解决方法
推荐用下面这套简化分工：

1. `main`：只放稳定可发布版本，不做日常开发
2. `dev`：日常集成分支，功能先合到这里联调
3. `feature/*`：一个功能一个分支，做完即合并、合并后尽快删除
4. `fix/*`：一个问题一个分支，小而快，避免和新功能混写

推荐日常流程：

```powershell
git checkout dev
git pull origin dev
git checkout -b feature/platform-api
```

开发完成后先回到 `dev`：

```powershell
git checkout dev
git pull origin dev
git merge feature/platform-api
git push origin dev
```

确认 `dev` 稳定后再进入 `main`：

```powershell
git checkout main
git pull origin main
git merge dev
git push origin main
```

### 后续注意

1. 一个分支只做一件事。
2. 功能分支和修复分支都应保持短生命周期。
3. 合并完成后及时删除无用分支。
4. 当前项目阶段不建议过早引入过重的 release 流程，先把 `main + dev + feature/fix` 用干净最重要。

## 问题 3：第三步“服务器基础环境准备”容易让人误以为要在服务器上手工创建开发文件

### 日期
2026-04-16

### 背景
文档前面已经明确了“业务代码不在 Debian 服务器上直接编写”，但第三步如果只写“创建统一部署目录”，容易让阅读者继续联想到要在服务器上手工创建源码目录和开发文件。

### 问题
第三步是否包含在服务器上直接创建 `main.py`、`requirements.txt`、`app/api/v1` 这类开发文件和源码目录的步骤？

### 原因
“创建部署目录”和“创建业务源码目录”这两件事在字面上很像，但职责完全不同。如果文档不把边界写死，执行时就容易回到旧习惯：先在服务器上搭一套代码骨架，再和本地仓库分开维护。

### 解决方法
需要在第三步明确写清楚：

1. 第三步只准备操作系统、运行目录、权限和环境依赖。
2. 不在服务器上手工创建业务源码文件。
3. 业务源码只能通过本地开发后提交 GitHub，再由服务器 `git clone` / `git pull` 进入。
4. 服务器上允许手工创建的内容主要是 systemd、Nginx、Jenkins、虚拟环境和运行期目录。

### 后续注意

1. 以后凡是文档里出现“mkdir”或“创建目录”，都要区分是运行目录还是业务源码目录。
2. 如果某个空目录需要被 Git 跟踪，应在本地仓库里通过 `.gitkeep` 或 `__init__.py` 解决，而不是在服务器上单独补建。

## 问题 4：需要把文档里所有“服务器上创建文件”的动作统一标注为运行配置还是业务源码

### 日期
2026-04-16

### 背景
即使第三步已经补充了“服务器不直接创建业务开发文件”的原则，文档后面仍然存在很多服务器侧写文件动作，例如 `sudo tee /etc/systemd/system/...`、`systemctl edit jenkins`、写入 `sudoers.d`、部署 Nginx 配置等。如果不统一标注类型，阅读时仍然容易混淆。

### 问题
如何让读者一眼看出：哪些是在服务器上允许创建的运行配置文件，哪些则属于绝不能在服务器上手工创建的业务源码文件？

### 原因
“创建文件”这个动作本身太宽泛。真正需要区分的是文件职责，而不是有没有创建动作：

1. 运行配置文件：可以在服务器上按部署需要生成或落地
2. 业务源码文件：必须由本地开发后通过 Git 同步进入服务器

### 解决方法
文档已补充两类改动：

1. 在第三步后新增对照表，集中列出哪些能手工建、哪些绝不能手工建。
2. 在 systemd、Nginx、sudoers、APT 源、SSH 配置等具体位置，统一标注为 `运行配置文件`。

### 后续注意

1. 以后文档里只要出现服务器侧写文件动作，都应补一个类型标注。
2. 默认优先问“这是运行配置还是业务源码”，再决定是否允许服务器手工创建。

## 问题 5：需要给整篇文档中的服务器动作统一固定标签，并补服务器动作分类总表

### 日期
2026-04-16

### 背景
在前面已经区分“运行配置文件”和“业务源码文件”后，文档仍然缺少统一视觉格式。不同位置虽然有解释，但读者需要逐段理解，不能快速扫读出哪些动作允许做、哪些要谨慎、哪些禁止做。

### 问题
如何把整篇文档里服务器侧的 `创建`、`写入`、`复制`、`编辑` 一类动作做成统一格式，并额外给出一张总表快速判断 `mkdir`、`tee`、`cp`、`git pull`、`pip install`、`systemctl` 的风险级别？

### 原因
如果没有统一标签，执行文档时容易只看到命令本身，而忽略动作类型和边界。运维动作和业务源码动作混在一起时，误操作概率会明显上升。

### 解决方法
文档已新增两套统一机制：

1. 固定格式标签：`服务器动作标签：允许做/谨慎做/禁止做 + 对象类型 + 动作命令`
2. 第三步后的服务器动作分类总表：集中说明 `mkdir`、`tee`、`cp`、`git pull`、`pip install`、`systemctl` 的默认分类和适用边界

### 后续注意

1. 以后新增服务器动作时，默认先补标签，再写命令。
2. 如果某条命令既可能安全也可能危险，应归到“谨慎做”，并写清允许场景与禁止场景。

## 问题 6：需要把服务器动作总原则前移，并统一成“动作标签 + 命令块 + 解释”三段式

### 日期
2026-04-16

### 背景

### 问题
如何把这套规则真正收口成全篇通用总原则，并避免标签和命令块混排导致格式错误？

### 原因
如果总原则放得太靠后，读者往往先开始执行命令，再回头理解标签体系。再加上标签没有完全统一到“标签 -> 命令 -> 解释”的三段式，就容易出现局部格式错乱。

### 解决方法
文档已进一步调整为：

1. 在第一步后新增“服务器动作执行总原则”作为统一入口。
2. 把服务器动作分类总表前移到总原则章节。
3. 第三步以后默认继承这套规则。
4. 修正了标签插进代码块内部的格式问题，并把关键服务器动作统一成“动作标签 + 命令块 + 解释”三段式。

### 后续注意

1. 后续新增服务器命令块时，优先检查三段式是否完整。
2. 标签必须放在代码块外，不能插进 bash、groovy、ini、nginx 等代码块内部。

## 问题 9：需要把 Jenkins 外部访问准备顺序前移到三块代码开发之前

### 日期
2026-04-16

### 背景
在实际执行时，当前项目先完成了 Jenkins Master 安装、插件安装和本机 `localhost:8080` 登录验证，但文档原本把 Nginx / HTTPS / Windows 外部访问放在后面的独立章节，排在“三块代码的开发与部署方式”之后。

### 问题
这样会导致执行者在进入 `platform-api`、`kpi-portal` 开发前，还要自己重新判断真实顺序，不适合直接照整篇文档往下走。

### 原因
对当前项目来说，更稳的真实顺序应该是：

1. 先把 Jenkins 的 `JENKINS_PREFIX=/jenkins` 配好。
2. 再把 Nginx 和 HTTPS 打通。
3. 再从 Windows 验证 `https://.../jenkins/` 可访问。
4. 最后再设置 Jenkins URL，并进入后续三块代码开发与部署。

如果这些步骤放在后面，执行者会先进入业务开发章节，后面又要回头补 Jenkins 外部入口配置。

### 解决方法
文档结构已调整为：

1. 把原 `第八步：Nginx 与 HTTPS 配置` 的核心内容并入 `第四步：Jenkins Master 部署` 后半段。
2. 在第四步中按实际顺序组织为：子路径配置、证书准备、Nginx 反代、Windows 访问验证、Jenkins URL 设置。
3. `第五步` 继续保留为“三块代码的开发与部署方式”，但现在它前面的环境入口顺序已经是可直接执行的。
4. 原“验证与日常运维”顺延为新的 `第八步`。

### 后续注意

1. 文档章节顺序应尽量贴近真实执行顺序，而不只是按功能模块分组。
2. 只要某一步是后续所有开发与部署的前置条件，就应尽量前移，避免读者边做边自己重排。 

## 问题 10：Nginx 的 `server { ... }` 配置块需要明确写清楚“这是文件内容，不是命令”

### 日期
2026-04-16

### 背景
在文档把 Jenkins 的 Nginx 与 HTTPS 配置前移到第四步后，执行顺序已经更合理，但 `4.8` 中那段很长的 `server { ... }` 仍然容易让人第一次阅读时不清楚到底要把它贴到哪里。

### 问题
执行者容易分不清：

1. 这是终端命令
2. 这是 Jenkins 里的配置
3. 还是这是一个需要保存成 `.conf` 文件的 Nginx 配置内容

### 原因
因为文档原本先给出了整段 `server { ... }`，后面才给 `cp` 命令。对熟悉 Nginx 的人来说这是自然的，但对第一次做这件事的人来说，缺少一句更直接的话：

“先在仓库里新建 `deploy/nginx/jenkins-kpi-platform.conf`，把下面内容写进去。”

### 解决方法
文档已补充得更直白：

1. 明确说明 `server { ... }` 是 Nginx 配置文件内容，不是命令。
2. 明确建议先在仓库新建 `deploy/nginx/jenkins-kpi-platform.conf`。
3. 明确说明仓库文件是维护位置，`/etc/nginx/sites-available/jenkins-kpi-platform.conf` 是服务器实际生效位置。
4. 再保留 `cp + ln -sf + nginx -t + restart` 作为落地启用步骤。

### 后续注意

1. 以后文档里只要出现大段 `ini`、`nginx`、`yaml`、`service` 内容，最好都补一句“这是文件内容，不是命令”。
2. 对第一次执行的人来说，“先在哪个文件里写，再复制到哪里”比单纯给配置块更重要。
